[02:00 - 03:00] SECTION 3: SYSTEM ARCHITECTURE & WEBHOOK CALLBACKS

"Now, let's talk about the system architecture. This project consists of three separate services to mirror a real-world enterprise campaign manager. 

First, we have the Next.js Frontend. Second, we have the Express CRM Backend powered by Prisma ORM and PostgreSQL. Third, we have a separate Channel Stub Service. 

When a campaign is launched, the CRM backend dispatches the messages to the Channel Stub. Instead of simply updating the database state statically, the Channel Stub simulates a real SMS/WhatsApp gateway. It introduces realistic network delays and sends webhook callbacks back to the CRM backend endpoint at '/api/receipts' with event statuses like 'delivered', 'opened', 'clicked', or 'failed'. The CRM backend processes these webhooks asynchronously and idempotently to update campaign metrics. This ensures our stats are perfectly synchronized with actual delivery logs."

[03:00 - 04:00] SECTION 4: CODE WALKTHROUGH

"Let's dive into the code. 

Here is our AI service layer at `gemini.ts`. It initializes the Google Generative AI SDK using the `gemini-2.5-flash` model. We pass a detailed system prompt instructing Gemini to parse natural language queries into a structured JSON schema containing segment filters (like min_orders, city, product_category), a suggested message template, and a friendly chat reply. If the Gemini API key is not supplied, it safely falls back to a rule-based parser so the system remains fully functional offline.

Next, look at the webhook handler in `receipts.ts`. To prevent duplicate event tracking from network retries, we run a transaction that does an idempotency check, progress-funnel validation (so we don't overwrite a click status with a delivery status), and updates campaign counters."

[04:00 - 05:00] SECTION 5: AI-NATIVE WORKFLOW & TRADEOFFS

"During this build, I leaned heavily on an AI-native workflow. I used Gemini to design and test the segment extraction prompt, generate the 1,200 mock order records in our seeding script, and troubleshoot a race condition in the async callback loop.

To build this quickly within a production-ready framework, I made a few explicit design tradeoffs:
1. For development ease, we bypass login authentication. In a production build, this would use JWT and organization scopes.
2. Webhook metrics are updated in Next.js using polling. At scale, this would be refactored to Server-Sent Events (SSE) or WebSockets.
3. Message queues are simulated in memory using timeouts; for scale, we would use Redis and BullMQ.
